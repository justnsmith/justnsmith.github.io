## Introduction
Taking ICS 314 has taught me a lot more about software engineering than just how to make web applications. While building web applications is a big part of it, I now see that the course was more about learning the fundamental concepts that apply to all areas of software development. A couple of ideas that stood out to me are: Agile Project Management, Configuration Management, and Coding Standards.

## Agile Project Management
Before taking ICS 314, I didn’t understand how software projects were managed. I kind of figured people just did the work and got it done. Agile flips this idea around as it's all about breaking work into smaller, manageable pieces and working in shorter cycles to adapt as things change. During the final project, we focused on something called Issue Driven Project Management (IDPM), which is essentially taking Agile principles and applying them to specific issues or tasks. Every issue created is tied to a clear goal and assigned to either a person or a group of people. This creates a structured plan, so we aren't just coding blindly.

What I liked about this approach is how flexible it is. I could think about what issues I wanted to work on, who I wanted to work on them with, and how long I expected them to take. This created a nice planned structure where I could see what everyone was working on. I could see myself using IDPM for something like a group coding project outside of web development or even a personal project like building a game. The whole idea of IDPM is about keeping things organized and being able to focus and adapt which makes sense no matter what the goal is.

## Configuration Management
When I first learned about Configuration Management, I thought it was just about using GitHub or just keeping track of files in general. However, it’s more about controlling changes to your codebase so you don’t accidentally mess everything up. There have been times when I have been coding and would decide to change something up and while I was changing this component it would mess up the rest of my code and I couldn’t go back to the old version so it was a big headache to me. Learning about Configuration Management was a big time saver and provided a safety net so I could feel more confident with experimenting with new ideas without having to worry about breaking the project.

This skill is useful beyond web development. I have used the techniques of configuration management in my ICS 311 class which is an algorithms class. We have these group assignments where we need to work together to produce an implementation of an algorithm along with separate problems. I’ve incorporated using GitHub along with separate branches to ensure that we have a safety net and that everything ties together nicely. Configuration management isn’t just tied to a specific tech stack; it's more like a universal way to make sure that you don’t lose your progress.

## Coding Standards
Coding Standards is one of those things that seem small and insignificant but make a huge difference. Coding standards are rules for writing code so that it’s consistent and easy to read. We used ESLint in this class, which is a tool that checks your JavaScript code against a set of rules and gives you errors if you messed up. At first, I thought it was annoying since I got a lot of warnings about tiny things such as not having a new line at the end of my file and I would also get errors pop up before I was done coding. But over time, I realized that it forces you to write cleaner code and this is important when you’re working with other people.

Beyond web development, I believe tools like ESLint would be helpful in any project. When I wrote C code in my ICS 212 class, we used Pylint as a coding standard. This made my code clearer and easier to read to not only my classmates but also the instructors. The point is, that having a consistent style will make your code easier to read, maintain and debug. This will also help everyone on the team stay on the same page since they’ll be familiar with your coding style rather than trying to understand something they aren’t used to.

## Final Thoughts
The most important thing I’ve learned is that software engineering is way more than just coding. It involves the tools, techniques, and processes that make coding more efficient and sustainable. It doesn’t matter if it's managing tasks with Agile, keeping order with Configuration Management, or following Coding Standards so your code is well put together, these skills will stick with me no matter what kind of software I will be working on in the future.
